The "manage.py" File

All Django projects come with a manage.py file at project root. In short, this file effectively acts as the "entrypoint" for any Django project.

For example, manage.py can be used to:

  • Create new project app.
  • Update the database, based on project models.
  • Run a local instance of the project server.
  • Run any custom commands you write.

Using "manage.py"

The manage.py file is technically just a "standard Python file". It is run the same way as any other Python file.

So like any other Python file, first you'll need to load your desired Virtual Environment.

Once your virtual environment is loaded, you can simply run python <project_root>/manage.py. This will give a list of all possible commands that the given project manage.py file accepts.

From here, you can simply run the above command again, but with the desired command as the first argument. For example, to run the project server, use python <project_root>/manage.py runserver.

Workspace and "manage.py"

Since "manage.py" is simply an entrypoint file to execute project code, there's no benefit to having multiple manage.py files in a given project. Furthermore, since CAE Workspace is meant to have subprojects hook into it, it makes sense to only have a manage.py file in Workspace itself.

To rephrase, there is no benefit to having manage.py files exist within subprojects. In fact, it's likely detrimental, as a subproject manage.py likely won't have access to the same "generalized logic" that Workspace provides.

Thus, when working with Workspace or an associated subproject, you should always use the manage.py file found in the Workspace root folder.
